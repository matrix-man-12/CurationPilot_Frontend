Backend Integration & System Design Guide

For: Backend & Infrastructure Teams
Version: 1.0.0

1. Local Loopback & CORS Configuration (Local-Only Execution)

Since this platform runs entirely locally on the user's own machine, the frontend (Vite dev server) and backend API server run on different local ports during development:

  • Frontend dev server URL: http://localhost:5173
  • Backend API URL: http://localhost:8080/api
Cross-Origin Resource Sharing (CORS) Requirement

Because the browser blocks Cross-Origin HTTP requests by default, the backend API server (running on port 8080) MUST enable CORS to explicitly authorize incoming requests from http://localhost:5173. If CORS headers are omitted, all frontend fetches, status polling, and control calls will be blocked by the browser.

CORS Header Requirements

The backend must respond to all preflight and API requests with these headers:

Access-Control-Allow-Origin: http://localhost:5173
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization

2. Single Page Application (SPA) Routing Fallback

The CurationPilot React frontend uses standard HTML5 client-side history routing for URL paths such as /history and /logs.

Server Rewrite Requirement

When serving the static React build, the production server (Nginx, Express, etc.) must rewrite any requests that do not map to static files or API endpoints to serve index.html. If this fallback is omitted, users refreshing on pages other than the homepage will encounter a 404 Not Found error.

A. Nginx Server Configuration

server {
    listen 80;
    server_name automation.curationpilot.com;
    root /var/www/curationpilot/dist;
    index index.html;

    # Static assets fallback rewrite
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Proxy REST API requests to backend
    location /api/ {
        proxy_pass http://localhost:8080/api/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}

B. Node.js Express Static Serve with CORS

const express = require('express');
const cors = require('cors');
const path = require('path');
const app = express();

// Enable CORS for local frontend dev server
app.use(cors({
  origin: 'http://localhost:5173',
  methods: ['GET', 'POST', 'OPTIONS']
}));

// Serve API endpoints
app.use('/api', apiRouter);

// Serve static compiled assets
app.use(express.static(path.join(__dirname, 'dist')));

// Fallback all routes to index.html for React routing
app.get('*', (req, res) => {
  res.sendFile(path.join(__dirname, 'dist', 'index.html'));
});

app.listen(8080);

3. Playwright Automation Runner Design

To support the dashboard's status controls and log streams, the backend Playwright run controller should adhere to the following machine layout:

A. Execution State Machine

stateDiagram-v2 [*] --> queued : "POST /api/executions (returns exec_id)" queued --> running : "Worker thread allocated" running --> running : "Poll updates logs & progress" running --> paused : "POST /api/executions/:id/pause" paused --> running : "POST /api/executions/:id/resume" running --> waiting_for_user : "Playwright hits HITL node" waiting_for_user --> running : "POST /api/executions/:id/approve" waiting_for_user --> failed : "POST /api/executions/:id/reject" running --> completed : "Poll detects completed" running --> failed : "Poll detects failed" running --> cancelled : "POST /api/executions/:id/cancel" paused --> cancelled : "POST /api/executions/:id/cancel" completed --> [*] failed --> [*] cancelled --> [*]

B. State Descriptions & Requirements

Status Playwright Worker Thread Requirements
queued Placed in an in-memory queue. No browser instance is launched yet.
running Allocated worker instance, browser context opened, executing automation step script loops.
paused Worker thread execution suspended. **Keep browser context open.** Use Promise resolvers to block progression.
waiting_for_user HITL validation node reached. Halted awaiting user approval or rejection HTTP calls.
completed Task completed (exit code 0). Clean up assets, close browser, serialize output result object.
failed Script crashed, timed out, or user rejected HITL. Store error message and screenshot path.
cancelled Receive cancel payload. Close browser immediately, free workers, set status.

4. Bulk CSV Execution Mapping

Bulk execution requests are submitted as a single POST body with an array of objects in the parameters:

{
  "skillId": "skill_001",
  "parameters": {
    "isBulk": true,
    "rowCount": 3,
    "csvRows": [
      { "portal_url": "https://company-a.com", "vendor_ids": ["V1"] },
      { "portal_url": "https://company-b.com", "vendor_ids": ["V2"] },
      { "portal_url": "https://company-c.com", "vendor_ids": ["V3"] }
    ],
    "portal_url": "https://company-a.com",
    "vendor_ids": ["V1"]
  }
}

Backend Requirements for Bulk Execution

  • Batch Processing: Process the csvRows sequentially or load them into parallel worker slots according to server resource limits.
  • Log Output: Format the stdout log strings to include row indicators: "[Row 2/3] Automating browser search...".
  • Progress Scale: Set totalSteps proportional to the size of the batch (e.g. rowCount * stepsPerRow), so progress percentages advance smoothly.

5. Log Serialization Format

Logs streamed inside the execution status must follow this exact layout:

"logs": [
  {
    "timestamp": "2026-06-15T10:30:02.123Z",
    "level": "info",
    "message": "Initializing browser context..."
  },
  {
    "timestamp": "2026-06-15T10:30:05.456Z",
    "level": "warn",
    "message": "Execution paused by user"
  }
]
  • Timestamps: ISO 8601 string including milliseconds to maintain order during sorting.
  • Levels: Restrict to info, warn, error, or debug.
  • Storage: Store log arrays associated with the execution ID in a database (e.g. MongoDB or PostgreSQL) so history views can query logs of completed runs.

6. Control API Contract Requirements

All control and HITL endpoints must update the running state machine and return the fully-updated execution status object in the response body (matching the structure of GET /api/executions/:id).

Visual Delay Avoidance
Returning the full execution record immediately ensures that the React frontend updates progress bars and logs instantaneously upon user click, preventing a 1.5s visual delay.
