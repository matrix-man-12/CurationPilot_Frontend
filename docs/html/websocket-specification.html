WebSocket & Webhook Specification

Status: Draft
Version: 1.0.0

Overview

Once the backend incorporates real-time communications support, the frontend will adopt WebSockets to stream execution events and Playwright action logs directly, deprecating short-polling. Outbound integrations will be handled via backend Webhooks to notify external user endpoints upon completion or failure.

WebSocket Connection

Connection Endpoint

Development:

ws://localhost:8080/ws/executions/:executionId

Production:

wss://api.curationpilot.com/ws/executions/:executionId

Handshake Authentication (Future)

When user verification is implemented, authentication parameters will be attached directly to the connection query string during handshake:

ws://host/ws/executions/:executionId?token=<jwt_token>

Connection Lifecycle

  1. Frontend triggers POST /api/executions and receives the tracking executionId.
  2. Frontend establishes a WebSocket connection targeting /ws/executions/:executionId.
  3. Backend accepts connection and starts streaming script log and progress events.
  4. WebSocket connection terminates automatically when execution lands in a terminal state (completed, failed, or cancelled).

Server ➔ Client Event Envelopes

All structured events streamed from the server follow this uniform structure:

{
  "event": "execution.step",
  "timestamp": "2026-06-15T10:30:05Z",
  "data": { ... }
}

Event Types

1. execution.started

Sent as soon as Playwright executes the script file.

{
  "event": "execution.started",
  "timestamp": "2026-06-15T10:30:02Z",
  "data": {
    "executionId": "exec_abc123",
    "totalSteps": 8
  }
}

2. execution.step

Sent whenever a Playwright step completes successfully.

{
  "event": "execution.step",
  "timestamp": "2026-06-15T10:30:05Z",
  "data": {
    "executionId": "exec_abc123",
    "step": 3,
    "totalSteps": 8,
    "stepName": "Navigating to invoice list",
    "percentage": 37,
    "screenshotUrl": "/screenshots/exec_abc123_step3.png"
  }
}

3. execution.log

Real-time stdout log line emitted by the automated script runner.

{
  "event": "execution.log",
  "timestamp": "2026-06-15T10:30:05Z",
  "data": {
    "executionId": "exec_abc123",
    "level": "info",
    "message": "Login successful, navigating to invoice page"
  }
}

Supported log levels: "debug", "info", "warn", "error".

4. execution.completed

Sent upon complete run success. Terminates connection.

{
  "event": "execution.completed",
  "timestamp": "2026-06-15T10:32:15Z",
  "data": {
    "executionId": "exec_abc123",
    "duration": "2m 13s",
    "result": {
      "summary": "Successfully extracted 24 invoice line items",
      "outputUrl": "/downloads/exec_abc123_output.csv",
      "data": {
        "invoicesFound": 24,
        "vendorsProcessed": 3,
        "totalAmount": 45230.50
      }
    }
  }
}

5. execution.failed

Sent when the runner fails or is rejected. Terminates connection.

{
  "event": "execution.failed",
  "timestamp": "2026-06-15T10:31:45Z",
  "data": {
    "executionId": "exec_abc123",
    "error": {
      "code": "NAVIGATION_TIMEOUT",
      "message": "Timed out waiting for invoice list page to load",
      "step": 3,
      "screenshotUrl": "/screenshots/exec_abc123_error.png"
    }
  }
}

6. execution.cancelled

Sent when cancellation is confirmed by the backend thread. Terminates connection.

{
  "event": "execution.cancelled",
  "timestamp": "2026-06-15T10:31:00Z",
  "data": {
    "executionId": "exec_abc123",
    "cancelledAt": "2026-06-15T10:31:00Z"
  }
}

Client ➔ Server Event Envelopes

1. execution.cancel

Allows the client to request immediate execution abortion over the WebSocket pipe.

{
  "event": "execution.cancel",
  "data": {
    "executionId": "exec_abc123"
  }
}

Reconnection Strategy

Should the WebSocket channel drop unexpectedly during execution, the client executes an exponential backoff reconnect attempt:

  1. The frontend detects connection close state via ws.onclose.
  2. Wait 1 second, then invoke a reconnection handshake.
  3. On success, verify state by querying GET /api/executions/:executionId, pulling down any logs that were generated while disconnected.
  4. Resume standard event listening.
  5. If unsuccessful, double the retry delay: 1s ➔ 2s ➔ 4s ➔ 8s ➔ max 30s.
  6. After 10 failed reconnection attempts, fall back to the REST short-polling loop automatically.

Webhooks (Outbound Notifications)

Allows external applications (third-party tools, databases) to listen to completed automation notifications.

Configuration Payload

{
  "webhookUrl": "https://your-system.com/webhooks/curationpilot",
  "events": ["execution.completed", "execution.failed"],
  "secret": "whsec_..."
}

Webhook Request Body

{
  "event": "execution.completed",
  "timestamp": "2026-06-15T10:32:15Z",
  "data": {
    "executionId": "exec_abc123",
    "skillId": "skill_001",
    "skillName": "Extract Invoice Data",
    "status": "completed",
    "duration": "2m 13s",
    "result": {
      "summary": "Successfully extracted 24 invoice line items"
    }
  },
  "signature": "sha256=..."
}

Signature Verification

The backend signs webhook payloads using HMAC-SHA256 with a pre-shared secret. Receiving endpoints should verify the signature key/header to guarantee that the request payload originated from CurationPilot.
