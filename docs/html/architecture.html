Frontend Architecture

Status: Completed
Updated: June 15, 2026

Component Tree

The CurationPilot frontend is structured as a component-driven React application. Below is the primary visual layout hierachy:

App (Root Layout, View Toggler)
├── AppHeader
│   ├── Logo + Brand Title
│   └── Navigation Tabs (Chat / History / Logs views)
│
├── ChatView (Active workspace)
│   └── ChatContainer
│       ├── MessageList (Timeline container)
│       │   ├── System Message: SkillSelector (Nested, expandable category list)
│       │   ├── System Message: ParameterForm (Form inputs + CSV Dropzone tab panels)
│       │   │   ├── Manual Form Inputs (Rendered by type)
│       │   │   └── CSV preview table
│       │   ├── User Message: Grid summary cards for parameters submitted
│       │   └── System Message: ExecutionStatus (Progress bar, action toolbar, logs, HITL dialog)
│       └── (Note: All user configuration and triggers are contextual bubble items)
│
├── HistoryView (Past logs and execution lookup)
│   ├── History Search Bar
│   └── SessionCardList (Checkbox selection for bulk deletion, clicks reload session)
│
└── LogsView (Streaming live console log tracker)
    ├── Log control toolbar (filter dropdowns, clear logs, pause/resume browser feeds)
    └── Monospace logs terminal block

State Management Architecture

Global app state is stored centrally in a React Context (AppContext) with local storage bindings:

AppContext (State Provider)
│
├── currentView: "chat" | "history" | "logs"
├── activeSession: {
│     id: String,
│     skillId: String,
│     skillName: String,
│     status: "idle" | "configuring" | "executing" | "completed" | "failed",
│     parameters: Object,
│     executionId: String,
│     messages: Array [
│       { id, sender: "user"|"system", type: "text"|"form"|"execution"..., text, meta: { logs, result } }
│     ]
│   }
├── sessions: Array (Historical session configurations list)
│
└── Dispatch Actions:
    ├── SET_VIEW: Switch active panels
    ├── START_NEW_SESSION: Clear active and select a new skill
    ├── SELECT_SKILL: Mount skill parameters schema
    ├── SUBMIT_PARAMETERS: Store input payload, transition to executing status
    ├── SET_EXECUTION_ID: Bind Playwright execution runner session ID
    ├── UPDATE_LOGS: Stream active logs array to active message meta
    ├── UPDATE_EXECUTION: Update progress metrics and status strings
    ├── COMPLETE_EXECUTION: Populate final duration and report summary
    ├── FAIL_EXECUTION: Bind error description and halt updates
    └── LOAD_SESSION: Pull past session from history array and replace active context

Data Flow

Interaction data flows in a unidirectional pattern:

User Interaction Triggered Dispatch Action State Updates UI Re-render Event
Select Skill in Dropdown SELECT_SKILL Binds skillId and parameters template schema. Mounts dynamic ParameterForm.
Click "Run Skill" button SUBMIT_PARAMETERS Injects parameter object, App state ➔ "executing". Adds user parameter receipt bubble and starts ExecutionStatus.
HTTP Polling ticks (1.5s) UPDATE_EXECUTION, UPDATE_LOGS Updates active session logs arrays and step percentages. Scrolls logs into view, updates progress bar indicators.
User clicks Pause/Cancel Sends API request, updates local state on success. Transitions status to paused or failed. Freezes progress indicators, displays control states.
Select Session card in History LOAD_SESSION Pulls the corresponding record matching session ID. Restores chat timeline, inputs, logs, and statuses exactly as they were left.

Key Design Decisions

  1. Native HTML5 Routing: Transitioning between the three views (/, /history, /logs) updates the URL path natively using the HTML5 History API (window.history.pushState and popstate listeners in AppContext.jsx). This preserves the active view when refreshing the page without requiring an external routing framework.
  2. Single Active Session: One active chat is handled at a time. The active session holds parameter configurations, log streams, execution status, progress metrics, and transaction validation flags.
  3. Mock API Layer: mockApi.js mirrors the real backend API interfaces. Real-world parameters and response wrappers match perfectly, allowing a clean swap when the real backend REST endpoints are ready.
  4. localStorage Persistence:
    • Session Store: All past and active chat sessions (including in-chat execution logs inside message meta properties) are saved to localStorage, surviving page refreshes and view switches.
    • Live Logs Store: Console operations logged in the Logs tab are buffered in localStorage to avoid clearing them when transitioning views.
    • Execution Tracker: The active execution store maps IDs to current operational states in localStorage, allowing the client component to automatically resume short-polling when mounting.
  5. CSS Custom Properties: Custom design tokens are organized in variables.css, facilitating a highly custom, responsive warm-pastel visual system without depending on external CSS frameworks.
